Permission 权限系统

源码位置：packages/opencode/src/permission/next.ts (287 行)

概述

Permission 系统是 OpenCode 的安全核心——一个白名单模型。 默认行为是询问用户，只有被明确允许的操作才会静默执行。 每个工具在执行前都必须过权限检查，没有绕过后门。

权限规则结构

type PermissionRuleset = Record<string, PermissionRule>

// PermissionRule 可以是:
//   简单字符串: "allow" | "ask" | "deny"
//   或模式对象: Record<glob, "allow" | "ask" | "deny">

// ═══════ 示例 ═══════

{
  // 简单规则 — 对整个权限类型生效
  "read": "allow",              // 所有文件读取自动允许
  "bash": "ask",                // 所有 bash 命令都要确认
  "webfetch": "ask",            // 网络请求要确认

  // 模式规则 — 通配符匹配
  "edit": {
    "*": "allow",               // 默认允许编辑
    ".env*": "deny",            // 禁止编辑 .env 文件
    "*.lock": "ask",            // 锁文件需要确认
    "node_modules/**": "deny",  // 禁止编辑 node_modules
  },

  // 外部目录
  "external_directory": {
    "*": "ask",                 // 默认询问
    "/tmp/*": "allow",          // 允许 /tmp
    "~/Desktop/*": "deny",      // 禁止桌面
  }
}

// ★ 评估顺序: 从上到下, 最后匹配的胜出
// 这意味着: 更具体的规则放在后面, 可以覆盖通配符

通配符匹配


  支持的模式语法:

  模式              匹配                    示例
  ──────────────────────────────────────────────────────
  "*"               任意字符串              一切
  "*.ts"            以 .ts 结尾            app.ts, index.ts
  "src/*"           src/ 直接子文件        src/app.ts
  "src/**"          src/ 所有后代          src/a/b/c.ts
  ".env*"           以 .env 开头           .env, .env.local
  "~/Desktop/*"     Home 目录下            /Users/me/Desktop/x
  "npm *"           npm 开头的命令         npm install, npm test
  "git checkout *"  git checkout 命令      git checkout main

  路径展开:
  "~/"  → "/Users/username/"
  相对路径 → 基于项目根目录解析为绝对路径

错误类型


  权限系统的三种错误:

  ┌────────────────────────────────────────────────────────────┐
  │  RejectedError — 用户主动拒绝                               │
  │                                                            │
  │  触发: 用户在权限确认框点击 "No"                             │
  │  效果:                                                     │
  │    → Agent 循环终止 (blocked = true)                       │
  │    → 取消该 Session 所有待确认权限                           │
  │    → 用户可附带反馈文本 ("请用另一种方式")                    │
  │    → feedback 作为工具错误返回给 LLM                        │
  └────────────────────────────────────────────────────────────┘

  ┌────────────────────────────────────────────────────────────┐
  │  CorrectedError — 用户拒绝但给了修正建议                    │
  │                                                            │
  │  触发: 用户修改了工具参数 (如修改文件路径)                   │
  │  效果:                                                     │
  │    → 工具执行被跳过 (但循环不终止)                          │
  │    → LLM 看到错误信息 + 用户的修正建议                      │
  │    → LLM 可以用修正后的参数重新调用工具                      │
  └────────────────────────────────────────────────────────────┘

  ┌────────────────────────────────────────────────────────────┐
  │  DeniedError — 规则自动拒绝 (非交互)                        │
  │                                                            │
  │  触发: 规则集中匹配到 "deny"                                │
  │  效果:                                                     │
  │    → 工具执行被跳过 (不询问用户)                            │
  │    → LLM 看到 "Permission denied" 错误                     │
  │    → LLM 可以尝试其他方式                                   │
  └────────────────────────────────────────────────────────────┘

"Always" 规则的传播


  用户点击 "Always" 时发生了什么:

  1. 工具请求:
     ctx.ask({
       permission: "bash",
       patterns: ["npm install lodash"],  ← 具体命令
       always: ["npm install *"],         ← "Always" 时保存的更宽泛模式
     })

  2. 用户选择 "Always":
     → 保存规则: { bash: { "npm install *": "allow" } }
     → 注意: 保存的是 "npm install *" 而不是 "npm install lodash"
     → 这样后续 "npm install express" 也会自动允许

  3. 同 Session 中的待确认权限自动解决:
     → 如果有其他 "npm install xxx" 正在等待用户确认
     → 这些请求会被新规则匹配 → 自动允许

  BashArity 模式生成:
  ┌──────────────────────────────────────────────┐
  │  命令                    always 模式          │
  │  ──────────────────────────────────────────── │
  │  "npm install lodash"  → "npm install *"     │
  │  "git checkout main"   → "git checkout *"    │
  │  "touch file.txt"      → "touch *"          │
  │  "rm -rf build"        → "rm *"             │
  │  "cd src && ls"        → "cd * && ls *"     │
  │  "echo hello"          → "echo *"           │
  └──────────────────────────────────────────────┘

设计要点

  • 白名单模型 — 默认安全 — 没有规则时默认 "ask"， 不会静默执行任何危险操作。只有用户明确允许的才会自动放行。
  • 最后匹配胜出 — 规则覆盖 — 规则按声明顺序评估，后面的覆盖前面的。 这让你可以先设 *: "allow" 再设特例。
  • Always 保存更宽泛的模式 — 用户允许 npm install lodash 时， 保存的是 npm install *。平衡了便利性和安全性。
  • 四层合并覆盖 — Session → Agent → Config → DB， 子 Agent 可以有更严格的限制，用户配置可以放宽默认值。
  • 反馈机制让 LLM 能学习 — 用户拒绝时可以说"请用另一种方式"， LLM 看到反馈后可以调整策略（换个命令、换个文件路径）。
