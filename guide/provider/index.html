概述

Provider 系统将 15+ LLM 提供商统一在同一个接口下。 但这不只是"换个 API Key"那么简单——每个 Provider 都有自己的消息格式怪癖、 Tool ID 限制、缓存机制、认证方式。Provider 层的核心工作就是抹平这些差异。

Model 数据结构 — 完整字段

type Provider.Model = {
  // ═══════ 标识 ═══════
  id: string              // 内部 ID e.g. "claude-opus-4"
  providerID: string      // Provider ID e.g. "anthropic"

  api: {
    npm: string           // SDK 包名 e.g. "@ai-sdk/anthropic"
    id: string            // API 模型 ID e.g. "claude-opus-4-20250514"
  }

  // ═══════ 能力声明 ═══════
  capabilities: {
    temperature: boolean  // 支持 temperature 参数?
    reasoning: boolean    // 支持 extended thinking?
    vision: boolean       // 支持图片输入?
    output_format: boolean // 支持结构化输出 (JSON schema)?
    computer_use: boolean // 支持计算机使用?
  }

  // ═══════ 上下文限制 ═══════
  limit: {
    context: number       // 总上下文窗口 e.g. 200000
    input?: number        // 最大输入 token (可选, 某些 Provider 分别限制)
    output: number        // 最大输出 token e.g. 16384
  }

  // ═══════ 成本追踪 ═══════
  cost?: {
    input: number         // 每百万 input token 美元 e.g. 15.0
    output: number        // 每百万 output token 美元 e.g. 75.0
    cache: number         // 缓存读取价格 e.g. 1.5
  }

  // ═══════ Provider 特定 ═══════
  options?: Record<string, any>       // e.g. { thinking: { budget: 10000 } }
  variants?: Record<string, Variant>  // 模型变体
  headers?: Record<string, string>    // 自定义请求头
}

  // 详细的 Provider 创建流程:
  function createProvider(providerID, config) {

    // ★ 第 1 步: 解析 API Key
    // 优先级: 环境变量 > 配置文件 > 插件 auth hook
    const apiKey = process.env.ANTHROPIC_API_KEY
                ?? config.api_key
                ?? await Plugin.auth({ provider: providerID })

    // ★ 第 2 步: 创建 SDK 实例
    const sdkCreate = await import(config.npm ?? defaultNpm)
    // e.g. import { createAnthropic } from "@ai-sdk/anthropic"

    const provider = sdkCreate({
      apiKey,
      baseURL: config.base_url,
      headers: {
        ...config.headers,
        // ★ OpenCode 自定义 header
        "x-opencode-version": VERSION,
      },
    })

    // ★ 第 3 步: 获取 LanguageModel
    return provider.languageModel(model.api.id)
    // e.g. provider.languageModel("claude-opus-4-20250514")
  }

缓存控制策略 — 省钱的关键


  ═══════ 为什么要缓存? ═══════

  不带缓存:
    每次 LLM 调用: System Prompt (4000 tokens) + 历史消息 (10000 tokens)
    = 14000 input tokens × $15/M = $0.21

  带缓存:
    首次: 14000 tokens → $0.21 + 缓存写入费
    后续: 缓存命中 14000 tokens × $1.5/M = $0.021  ← 90% 折扣!

  ═══════ 缓存标记策略 ═══════

  对话消息数组:
  ┌──────────────────────────────────────────────────┐
  │ system[0]: Agent Prompt          ← 缓存标记 ★   │  不变的部分
  │ system[1]: 环境信息 + 上下文     ← 不缓存       │  每次可能变
  │──────────────────────────────────────────────────│
  │ msg[0]: user "开始项目..."                       │
  │ msg[1]: assistant "好的..."                      │
  │ msg[2]: user "修改登录..."                       │
  │ msg[3]: assistant "我来看看..."                   │  中间消息
  │ ...                                              │  不标记缓存
  │ msg[N-2]: user "运行测试"        ← 缓存标记 ★   │
  │ msg[N-1]: assistant "测试结果..."  ← 缓存标记 ★  │  最后 2 条
  └──────────────────────────────────────────────────┘

  为什么只标记这几个?
  - System Prompt: 每次调用完全相同 → 缓存命中率 100%
  - 最后 2 条: 下一轮调用时大概率还在 → 缓存命中率高
  - 中间消息: 不标记 → 避免缓存条目过多导致命中率下降

  ═══════ 各 Provider 的缓存格式 ═══════

  Anthropic:
    providerOptions: { anthropic: { cacheControl: { type: "ephemeral" } } }

  Bedrock:
    providerOptions: { bedrock: { cachePoint: { type: "default" } } }

  OpenRouter:
    providerOptions: { openrouter: { cacheControl: { type: "ephemeral" } } }

  Copilot:
    providerOptions: { copilot: { copilot_cache_control: { type: "ephemeral" } } }

  // ★ 统一封装:
  const CACHE_OPTIONS = {
    anthropic:        { cacheControl: { type: "ephemeral" } },
    openrouter:       { cacheControl: { type: "ephemeral" } },
    bedrock:          { cachePoint: { type: "default" } },
    openaiCompatible: { cache_control: { type: "ephemeral" } },
    copilot:          { copilot_cache_control: { type: "ephemeral" } },
  }

成本计算


  ┌─────────────────────────────────────────────────────────────┐
  │  成本计算公式                                                │
  │                                                             │
  │  总成本 = input_cost + output_cost + cache_cost             │
  │                                                             │
  │  input_cost   = input_tokens   × (cost.input  / 1,000,000) │
  │  output_cost  = output_tokens  × (cost.output / 1,000,000) │
  │  reason_cost  = reason_tokens  × (cost.output / 1,000,000) │  ← 推理按 output 价
  │  cache_read   = cache.read     × (cost.cache  / 1,000,000) │  ← 缓存价
  │  cache_write  = cache.write    × (cost.input  / 1,000,000) │  ← 写入按 input 价
  └─────────────────────────────────────────────────────────────┘

  示例: Claude Opus 4 的一次调用

  ┌───────────────────────────────────────┐
  │  Token 明细:                          │
  │    Input:    5,000 tokens             │
  │    Output:   2,000 tokens             │
  │    Reasoning:  500 tokens             │
  │    Cache Read:3,000 tokens            │
  │    Cache Write:1,000 tokens           │
  │                                       │
  │  价格 (Claude Opus 4):                │
  │    Input:  $15.0 / M                  │
  │    Output: $75.0 / M                  │
  │    Cache:  $1.5 / M                   │
  │                                       │
  │  计算:                                │
  │    5000 × 15/M   = $0.075  (input)    │
  │    2000 × 75/M   = $0.150  (output)   │
  │     500 × 75/M   = $0.0375 (reason)   │
  │    3000 × 1.5/M  = $0.0045 (cache rd) │
  │    1000 × 15/M   = $0.015  (cache wr) │
  │                   ─────────           │
  │    总计:           $0.282              │
  └───────────────────────────────────────┘

  // 成本累积到 AssistantMessage:
  assistant.cost = 0.282
  assistant.tokens = {
    input: 5000, output: 2000,
    reasoning: 500,
    cache: { read: 3000, write: 1000 }
  }

工具结果中的媒体支持


  ═══════ 问题: 工具返回了图片, 哪些 Provider 能处理? ═══════

  比如 Read 工具读取了 screenshot.png, 返回 Base64 图片附件。
  但不是所有 Provider 都支持在 tool_result 中放图片:

  ┌──────────────────────────┬─────────────────────────┐
  │  Provider                │  tool_result 中的媒体    │
  ├──────────────────────────┼─────────────────────────┤
  │  Anthropic               │  ✓ 支持                 │
  │  OpenAI                  │  ✓ 支持                 │
  │  Amazon Bedrock          │  ✓ 支持                 │
  │  Vertex (Anthropic)      │  ✓ 支持                 │
  │  Google Gemini 3.x       │  ✓ 支持                 │
  │  Google Gemini 2.x       │  ✕ 不支持               │
  │  Mistral                 │  ✕ 不支持               │
  │  其他                    │  ✕ 不支持               │
  └──────────────────────────┴─────────────────────────┘

  ═══════ 不支持时的处理方式 ═══════

  不支持媒体的 Provider:
    tool_result 中的图片被提取出来 → 作为单独的 user message 注入

  原始消息:
    assistant: tool_call(read, screenshot.png)
    tool:      { output: "...", attachments: [image.png] }

  转换后 (不支持媒体的 Provider):
    assistant: tool_call(read, screenshot.png)
    tool:      { output: "..." }                          ← 移除图片
    user:      "Attached image(s) from tool result:"      ← 新增消息
               [image: screenshot.png]

设计要点

  • AI SDK 统一层是核心 — Vercel AI SDK 提供了统一的 streamText 接口， 但 OpenCode 在此之上加了 middleware（消息变换）和 Provider 特定处理， 处理了 AI SDK 没有覆盖的差异。
  • 消息规范化处理了 Provider 的怪癖 — Anthropic 拒绝空消息、 Claude 限制 Tool ID 字符、Mistral 要求 9 字符 ID + 不允许 tool→user 连续。 这些都是实战中发现的问题，不看源码你不会知道。
  • 缓存策略直接影响成本 — 合理的缓存标记可以将 API 费用降低 50-90%。 System Prompt + 最后 2 条消息的策略平衡了命中率和缓存条目数。
  • 成本透明 — 每次调用精确计算到 token 级别的费用， 用户可以看到每条消息、每个步骤的花费。
  • 零代码切换模型 — 只需改 config 中的 default_model， 所有 Provider 的差异由框架自动处理。
