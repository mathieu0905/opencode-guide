OpenCode Agent 框架 — 架构总览

OpenCode 是一个 生产级 AI Agent 开发框架，采用 TypeScript + Bun 实现后端， 提供完整的会话管理、工具调用、多模型支持和实时流式输出能力。 本指南将逐模块拆解其内部架构，帮助你理解如何从零构建一个 AI Agent 系统。

为什么要学习 OpenCode 架构？

如果你想构建一个 AI Agent（类似 Cursor、Claude Code、Windsurf 等）， 你会面对一系列工程问题：

  • 如何管理多轮对话的状态？（Session + Message 模型）
  • 如何让 AI 调用工具并获取结果？（Tool 系统 + Agent Loop）
  • 如何实时展示 AI 的思考过程？（SSE 事件流 + Delta 增量更新）
  • 如何防止 AI 执行危险操作？（Permission 权限系统）
  • 如何支持多个 LLM 提供商？（Provider 抽象层）
  • 如何在上下文过长时自动压缩？（Compaction 系统）

OpenCode 对以上每个问题都给出了完整的解决方案。学习它的架构， 等于学习了构建 AI Agent 的全套最佳实践。

核心数据流 — 一条消息的完整旅程

理解 OpenCode 架构的关键是理解这条数据流。当用户发送一条消息， 它会经过以下所有模块。下面用一个完整的例子说明：

第一梯队 — 核心数据模型

第二梯队 — 核心执行引擎

  1. Agent 执行循环 — 系统的心脏。Prompt 编排层 → Processor 流处理 → LLM 调用 的三层架构，以及 while(true) 循环如何驱动多轮工具调用。重点理解 Doom Loop 检测和自动压缩。
  2. Tool 工具系统 — Agent 与外部世界的桥梁。18+ 内置工具的接口定义、 执行流程、权限检查。学习如何开发自定义工具和 MCP 集成。

第三梯队 — 基础设施

  1. Event 事件总线 — 44 种事件类型、SSE 传输、前端消费模式。

  3. Storage 存储层 — SQLite WAL 模式、Drizzle ORM、事务副作用模式。

  1. Permission 权限系统 — 通配符规则引擎、交互式审批、Always 记忆机制。
  2. Config 配置系统 — 四层深度合并（系统→用户→项目→目录）。
  3. Plugin 插件系统 — Hook 机制、工具扩展、NPM 生态。

关键设计理念

1. Event-Driven（事件驱动）

所有内部状态变更都通过 Event Bus 广播，而不是直接通知消费者。 这意味着可以随时增加新的消费者（新的前端、日志系统、监控…）而不需要修改发布者代码。 前端通过 SSE 实时接收事件，实现零轮询的实时 UI。 后端使用 Database.effect() 确保事件在数据库事务提交后才发布，保证一致性。

2. Streaming-First（流式优先）

LLM 的输出不是等待完成后一次性返回，而是逐 token 通过 message.part.delta事件推送。用户在 AI 思考的同时就能看到输出。 对于长时间运行的工具调用，流式输出让用户知道系统没有卡住—— ToolPart 从 pending → running → completed 的状态变化全程可见。

3. Permission-First（安全优先）

这是一个 白名单模型——默认行为是询问用户， 只有被明确允许的操作才会静默执行。每个工具执行前都必须过权限检查，没有绕过后门。 用户选择 "Always" 后规则持久化到数据库，后续同类操作自动放行。

4. Loop-Based Agent（循环驱动）

Agent 不是简单的「请求 → 响应」，而是一个 while(true) 循环： LLM 输出 → 执行工具 → 结果反馈 → LLM 继续输出 → … 循环直到 LLM 认为任务完成（不再调用工具）或被外部中断。 这就是 AI 能完成复杂多步骤任务的原因——它可以在循环中不断调整策略。 配合 Doom Loop 检测和上下文自动压缩，防止循环失控。

5. Git-Native（Git 原生）

项目 ID 来自 Git 的第一个 root commit hash（同一仓库的不同 clone 共享同一 ID）。 Snapshot 系统基于 Git 对象数据库追踪文件变更。 Revert 功能通过 Git 快照恢复文件到任意时间点。 分支切换通过 .git/HEAD 文件监听实时感知。

6. Context-Based Isolation（上下文隔离）

每个项目目录拥有独立的 Instance 上下文（LSP 客户端、文件监听器、VCS 状态、插件实例）。 同时打开两个项目不会互相干扰。Instance 通过 AsyncLocalStorage 实现， 每个请求自动绑定到对应的项目上下文。
