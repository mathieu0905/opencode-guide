Tool 工具系统

源码位置：packages/opencode/src/tool/tool.ts (接口定义)

工具解析：packages/opencode/src/session/prompt.ts (resolveTools)

概述

Tool 系统是 Agent 框架中工程量最大的模块。它不只是"定义一个函数让 LLM 调用"—— 而是一套包含参数校验、权限检查、输出截断、插件 Hook、LSP 集成、 Tree-sitter 解析、9 层替换策略的完整框架。 每个工具都是一个精心设计的产品级组件。

Tool 接口 — 不只是 execute()

// tool.ts — 完整接口

type Tool.Info<Params extends ZodType, Metadata> = {
  id: string    // "bash", "read", "edit", "write", ...

  init: (ctx?: InitContext) => Promise<{
    // ★ description 直接影响 LLM 的调用决策
    // 写得好 → LLM 在正确时机调用
    // 写得差 → LLM 胡乱调用或从不调用
    description: string

    // ★ Zod schema → 自动转为 JSON Schema → 传给 LLM
    parameters: Params

    // 实际执行
    execute: (args, ctx: Context) => Promise<{
      title: string              // UI 显示标题
      metadata: Metadata         // UI 显示的元数据
      output: string             // ★ 返回给 LLM 的文本
      attachments?: FilePart[]   // 可选: 图片/PDF 附件
    }>

    // 自定义参数校验错误格式 (给 LLM 看的)
    formatValidationError?: (error: ZodError) => string
  }>
}

// ★ Tool.define() 自动包装:
// 1. Zod 参数校验 (execute 之前)
// 2. 输出截断 (execute 之后)
//    MAX_LINES = 2000, MAX_BYTES = 50KB

type Tool.Context = {
  sessionID: string
  messageID: string
  callID: string             // LLM 生成的唯一调用 ID
  abort: AbortSignal         // 用户中止信号
  agent: string              // 当前 Agent 名称
  messages: WithParts[]      // ★ 完整对话历史 (工具可以查看上下文)

  extra: {
    model: Provider.Model    // 当前使用的模型
    bypassAgentCheck: boolean
  }

  // ★ 请求权限 — 工具执行前必须调用
  ask: (req: {

    title?: string
    metadata?: any

工具组装流程 — resolveTools()

resolveTools() 不是简单地返回工具列表。 每个工具都被包裹了权限检查、插件 Hook、输出截断、元数据更新：

// prompt.ts — resolveTools() 简化版

async function resolveTools(input) {
  const tools = {}

  // ═══════ 第 1 步: 加载内置工具 ═══════
  for (const item of await ToolRegistry.tools(model, agent)) {
    // Schema 转换: Zod → JSON Schema → Provider 规范化
    const schema = ProviderTransform.schema(model,
      z.toJSONSchema(item.parameters)
    )

    tools[item.id] = {
      description: item.description,
      inputSchema: jsonSchema(schema),

      // ★ 每个工具调用都经过这个包装器
      async execute(args, options) {
        const ctx = createContext(args, options)

        // 插件 Hook: 执行前
        await Plugin.trigger("tool.execute.before", {
          tool: item.id, sessionID, callID,
        }, { args })

        // 实际执行
        const result = await item.execute(args, ctx)

        // 插件 Hook: 执行后 (可修改结果)
        await Plugin.trigger("tool.execute.after", {
          tool: item.id, sessionID, callID, args,
        }, result)

        return result
      }
    }
  }

  // ═══════ 第 2 步: 加载 MCP 工具 ═══════
  for (const [key, mcpTool] of await MCP.tools()) {
    // MCP 工具同样包装: 权限 + 截断 + 插件
    mcpTool.execute = async (args, opts) => {
      await ctx.ask({ permission: key, patterns: ["*"] })
      const result = await originalExecute(args, opts)
      // 将 MCP 内容格式 (text/image/resource) → OpenCode 格式
      return { title: "", output: truncated, metadata: {} }
    }
    tools[key] = mcpTool
  }

  // ═══════ 第 3 步: 用户/Agent 禁用的工具 ═══════
  // UserMessage.tools 可以指定 { "todowrite": false }
  for (const [tool, enabled] of Object.entries(user.tools ?? {})) {
    if (enabled === false) delete tools[tool]
  }

  return tools
}

Bash 工具 — 最复杂的单个工具

Bash 工具不是简单的 exec(command)。它使用Tree-sitter 解析器分析命令 AST，提取路径信息做权限检查， 实现超时控制和进程树清理。

// bash.ts — 参数
{
  command: string      // 要执行的命令
  timeout?: number     // 超时 (默认 120000ms = 2 分钟)
  workdir?: string     // 工作目录 (默认: 项目目录)
  description: string  // 命令描述 (5-10 字)
}

// ═══════ Tree-sitter 命令解析 ═══════
// 为什么用 AST 解析而不是简单的字符串分割？
// 因为: "cd src && rm -rf build" 包含 2 个命令,
// 且 "rm -rf" 的参数 "build" 需要解析为路径

const tree = treeSitterBash.parse(params.command)
for (const node of tree.rootNode.descendantsOfType("command")) {
  const tokens = node.namedChildren.map(c => c.text)
  // tokens = ["cd", "src"] 或 ["rm", "-rf", "build"]

  // ★ 路径提取: 哪些命令的参数是路径？
  const PATH_COMMANDS = ["cd", "rm", "cp", "mv", "mkdir", "touch",
                         "chmod", "chown", "cat"]

  if (PATH_COMMANDS.includes(tokens[0])) {
    for (const arg of tokens.slice(1)) {
      if (arg.startsWith("-")) continue  // 跳过选项

      // 解析为绝对路径
      const resolved = await $`realpath ${arg}`.cwd(cwd).text()

      // ★ 检查是否在项目目录外
      if (!Instance.containsPath(resolved)) {
        externalDirs.add(path.dirname(resolved))
      }
    }
  }
}

// ★ 外部目录需要额外权限
if (externalDirs.size > 0) {
  await ctx.ask({
    permission: "external_directory",
    patterns: [...externalDirs].map(d => d + "/*"),
  })
}

// ★ Bash 命令本身需要权限
// 使用 BashArity 生成语义化的权限模式:
// "npm install lodash" → 权限模式 "npm install *"
// "git checkout main"  → 权限模式 "git checkout *"
// "touch file.txt"     → 权限模式 "touch *"
await ctx.ask({
  permission: "bash",
  patterns: [...commandTexts],
  always: [...BashArity.prefix(command) + "*"],
})
// bash.ts — 进程执行与控制

// 启动进程
const proc = spawn(params.command, {
  shell: Shell.acceptable(),  // bash/sh/pwsh (跨平台)
  cwd: workdir,
  env: {
    ...process.env,
    // ★ 插件可以注入环境变量
    ...(await Plugin.trigger("shell.env", { cwd }, { env: {} })).env,
  },
  stdio: ["ignore", "pipe", "pipe"],
  detached: true,  // 独立进程组 (方便清理)
})

// ★ 实时输出流: 每有新数据就更新 UI
proc.stdout.on("data", (chunk) => {
  output += chunk.toString()
  // metadata 实时更新 (UI 显示最后 30KB)
  ctx.metadata({
    title: params.description,
    metadata: { output: output.slice(-30000), exit: null },
  })
})

// ★ 三种终止场景
let timedOut = false, aborted = false

// 1. 超时
const timer = setTimeout(() => {
  timedOut = true
  Shell.killTree(proc)  // 杀掉整个进程树
}, timeout + 100)

// 2. 用户中止
ctx.abort.addEventListener("abort", () => {
  aborted = true
  Shell.killTree(proc)
})

// 3. 正常完成
await proc.exited

// 输出标签
if (timedOut) output += "\n<bash_metadata>(terminated after timeout)</bash_metadata>"
if (aborted) output += "\n<bash_metadata>User aborted</bash_metadata>"

Edit 工具 — 9 层替换策略

Edit 工具的核心难题：用户（LLM）提供的 oldString 可能与文件中的实际内容 有微妙差异（缩进不同、空白不同、转义不同）。OpenCode 设计了9 种递进的替换策略，从精确匹配到模糊匹配逐层尝试。

// edit.ts — 9 种替换器 (按顺序尝试, 第一个成功的胜出)

// 1. SimpleReplacer — 精确子串匹配
//    content.indexOf(search)
//    最快, 但要求完全一致

// 2. LineTrimmedReplacer — 逐行 trim 后匹配
//    文件中: "  const x = 1"  (2空格缩进)
//    搜索:   "    const x = 1"  (4空格缩进)
//    → trim 后都是 "const x = 1" → 匹配成功
//    → 替换时保留原始缩进

// 3. BlockAnchorReplacer — 首尾锚定 + Levenshtein 模糊匹配
//    要求: 搜索模式至少 3 行
//    逻辑:
//    - 第一行和最后一行作为 "锚点" (trim 后精确匹配)
//    - 中间行用 Levenshtein 距离计算相似度
//    - 单候选: 相似度 >= 0.0 (非常宽松)
//    - 多候选: 相似度 >= 0.3 (选最佳)
//    场景: 代码被 formatter 重排后, 首尾不变但中间空格变了

// 4. WhitespaceNormalizedReplacer — 所有空白归一化
//    将连续空白替换为单空格后匹配
//    "const   x  =  1" → "const x = 1"
//    场景: 代码格式化后空白变化

// 5. IndentationFlexibleReplacer — 忽略缩进级别
//    去除搜索模式和文件内容的基础缩进后匹配
//    场景: 代码被移到了不同的嵌套层级

// 6. EscapeNormalizedReplacer — 反转义匹配
//    将 \n → 换行, \t → Tab, \' → 单引号 等
//    场景: LLM 从字符串字面量中复制了代码, 带着转义字符

// 7. TrimmedBoundaryReplacer — 首尾空白裁剪
//    如果搜索模式有多余的首尾空白, 去掉后重试
//    场景: 复制粘贴时多了空行

// 8. ContextAwareReplacer — 上下文锚定
//    要求: 至少 3 行
//    - 第一行/最后一行作为上下文 (trim 后匹配)
//    - 50% 的中间行需要匹配
//    - 只返回第一个匹配
//    场景: 代码被部分修改, 但上下文没变

// 9. MultiOccurrenceReplacer — 多次出现
//    返回所有精确匹配的位置
//    配合 replaceAll: true 使用

// ═══════ 匹配验证 ═══════
for (const replacer of [Simple, LineTrimmed, BlockAnchor, ...]) {
  for (const candidate of replacer(content, oldString)) {
    const index = content.indexOf(candidate)
    if (index === -1) continue

    if (replaceAll) {
      return content.replaceAll(candidate, newString)
    }

    // ★ 必须是唯一匹配 (除非 replaceAll)
    const lastIndex = content.lastIndexOf(candidate)
    if (index !== lastIndex) continue  // 多处出现, 跳过

    return content.slice(0, index) + newString + content.slice(index + candidate.length)
  }
}
// 全部失败 → 报错: "Could not find oldString"
// 多处匹配 → 报错: "Found multiple matches, provide more context"
// edit.ts — 执行后的 LSP 诊断

// ★ 编辑完成后, 自动运行 LSP 检查
await LSP.touchFile(filePath, true)
const diagnostics = await LSP.diagnostics()
const errors = diagnostics[filePath]?.filter(d => d.severity === 1)

if (errors.length > 0) {

Read 工具 — 不只是 readFile()

// read.ts — 关键特性

// 参数
{
  filePath: string   // 文件或目录路径
  offset?: number    // 起始行号 (1-indexed)
  limit?: number     // 最多读取行数 (默认 2000)
}

// ═══════ 二进制文件检测 ═══════
function isBinary(filePath) {
  // 快速检查: 扩展名 (.zip, .exe, .class, .wasm, ...)
  if (BINARY_EXTENSIONS.has(ext)) return true

  // 深度检查: 读取前 4KB, 检查非打印字符比例
  const sample = readSync(filePath, 4096)
  const nonPrintable = sample.filter(byte =>
    byte === 0 || (byte < 32 && byte !== 9 && byte !== 10 && byte !== 13)
  )
  return nonPrintable.length / sample.length > 0.3
}

// ═══════ 图片/PDF → Base64 附件 ═══════
if (isImage || isPdf) {
  return {
    attachments: [{
      type: "file",
      mime: "image/png",  // 或 "application/pdf"
      url: `data:${mime};base64,${base64content}`,
    }],
    output: "File content returned as attachment",
  }
}

// ═══════ 文本文件逐行读取 ═══════
const rl = createInterface({
  input: createReadStream(filePath, { encoding: "utf8" }),
  crlfDelay: Infinity,  // 正确处理 \r\n
})

for await (const text of rl) {
  // 单行截断: > 2000 字符
  const line = text.length > 2000
    ? text.substring(0, 2000) + "... (line truncated)"
    : text

  // 字节预算: 总共不超过 50KB
  const size = Buffer.byteLength(line, "utf-8")
  if (bytes + size > 50_000) break

  raw.push(line)
}

// 输出格式 (带行号):
// 1: import React from 'react'
// 2: import { useState } from 'react'
// ...
// (Showing lines 1-50 of 200. Use offset=51 to continue.)

// ═══════ 指令注入 ═══════

  model: agent.model ?? parentModel,
  agent: agent.name,

  title: params.description,
  output: [
    `task_id: ${session.id} (for resuming if needed)`,

    "</task_result>",
  ].join("\n"),
}

输出截断引擎

// truncation.ts — 防止工具输出消耗过多 token

const MAX_LINES = 2000    // 最多 2000 行
const MAX_BYTES = 50_000  // 最多 50KB

async function truncateOutput(text, options, agent) {
  const lines = text.split("\n")

  // 在限制内 → 直接返回
  if (lines.length <= MAX_LINES && Buffer.byteLength(text) <= MAX_BYTES) {
    return { content: text, truncated: false }
  }

  // 超限 → 截断
  const preview = []
  let bytes = 0
  for (const line of lines) {
    const size = Buffer.byteLength(line)
    if (bytes + size > MAX_BYTES || preview.length >= MAX_LINES) break
    preview.push(line)
    bytes += size
  }

  // ★ 保存完整输出到文件 (7 天后自动清理)
  const filepath = path.join(TOOL_OUTPUT_DIR, uniqueID)
  await fs.writeFile(filepath, text)

  // ★ 提示 LLM 如何访问完整输出
  const hint = hasTaskTool(agent)
    ? `Full output saved to: ${filepath}\nUse Task tool to have explore agent process with Grep/Read`
    : `Full output saved to: ${filepath}\nUse Grep to search or Read with offset/limit`

  return {
    content: `${preview.join("\n")}\n\n...${removed} lines truncated...\n\n${hint}`,
    truncated: true,
    outputPath: filepath,
  }
}

工具选择逻辑 — 不是所有工具都可用

// registry.ts — 根据上下文动态决定工具集

function getTools(model, agent, config) {
  const tools = [
    // ═══════ 始终可用 ═══════
    "invalid",    // 处理 LLM 生成的无效工具名
    "bash",       // Shell 命令
    "read",       // 文件读取
    "glob",       // 文件搜索
    "grep",       // 内容搜索
    "edit",       // 文件编辑
    "write",      // 文件创建
    "task",       // 子任务
    "webfetch",   // 网页抓取
    "todowrite",  // 任务列表
    "skill",      // 技能调用
  ]

  // ═══════ 条件工具 ═══════

  // GPT 模型: 用 apply_patch 替代 edit + write
  // 因为 GPT 的 tool calling 对 apply_patch 格式更友好
  if (model.isGPT && !model.isOSS) {
    tools.remove("edit", "write")
    tools.push("apply_patch")
  }

  // 交互式提问: 只在有 UI 的客户端可用
  if (client === "app" || client === "cli" || client === "desktop") {
    tools.push("question")
  }

  // 搜索工具: 需要 Exa API 或 OpenCode 认证
  if (providerID === "opencode" || hasExaKey) {
    tools.push("websearch", "codesearch")
  }

  // 实验性工具
  if (config.experimental?.lsp) tools.push("lsp")
  if (config.experimental?.batch_tool) tools.push("batch")
  if (config.experimental?.plan_mode && client === "cli") tools.push("plan_exit")

  return tools
}

完整的工具执行流程

LLM 输出: tool_call { name: "edit", arguments: { filePath: "app.ts", ... } }
  │
  ▼
Processor (processor.ts):
  │  创建 ToolPart { status: "pending" }
  │  → Bus("message.part.updated") → SSE → 前端灰色卡片
  │  更新为 { status: "running" }
  │  → Bus("message.part.updated") → SSE → 前端黄色脉冲
  │
  ▼
resolveTools 包装器 (prompt.ts):
  │
  ├─ 1. Zod 参数校验
  │    args = schema.parse(rawArgs)
  │    失败 → 返回 formatValidationError() 给 LLM
  │
  ├─ 2. Plugin Hook: tool.execute.before
  │    插件可以修改参数或拒绝执行
  │    返回 null = 继续, 返回 { output } = 跳过执行
  │
  ├─ 3. Tool.execute(args, ctx)
  │    │
  │    ├─ ctx.ask() — 权限检查
  │    │    PermissionNext.ask({
  │    │      permission: "edit",
  │    │      patterns: ["src/app.ts"],
  │    │      ruleset: merge(agent.permission, session.permission),
  │    │    })
  │    │    → allow: 继续
  │    │    → ask: Bus("permission.asked") → 等待用户
  │    │    → deny: throw DeniedError
  │    │
  │    ├─ 实际执行 (e.g. EditTool: 9 层替换策略)
  │    │
  │    └─ LSP 诊断检查 (edit/write 专属)
  │

  │    if (output.length > 50KB || lines > 2000)
  │      → 保存到文件 → 返回截断版本
  │
  ├─ 5. Plugin Hook: tool.execute.after
  │    插件可以修改输出
  │
  └─ 6. 返回结果
       ToolPart { status: "completed", output, title, metadata }
       → Bus("message.part.updated") → SSE → 前端绿色卡片
       │
       ▼
  结果作为 tool_result 发回 LLM → 下一轮循环

设计要点

  • 9 层替换策略是 Edit 工具的核心竞争力 — 从精确匹配到 Levenshtein 模糊匹配， 处理了代码格式化、缩进变化、转义字符等常见问题。这是 LLM 编辑代码时最容易出错的环节。
  • Tree-sitter 解析让 Bash 工具有语义级理解 — 不是简单地匹配字符串， 而是解析 AST 提取命令和路径，实现精确的权限控制。
  • 输出截断 + 文件保存是 token 管理的关键 — 工具输出直接返回给 LLM 作为上下文， 不截断就会浪费 token。截断后保存到文件，LLM 可以用 Read/Grep 按需查看。
  • 每个工具都是 Permission-First — 即使是 Read 这种"安全"操作， 也要过权限检查。外部目录访问需要额外确认。没有后门。
  • LSP 集成让编辑有即时反馈 — Edit/Write 执行后自动运行 LSP 诊断， 如果引入了编译错误，LLM 能立即看到并修正。
  • 子 Agent (Task) 实现了真正的并行和隔离 — 子 Agent 在独立 Session 中运行， 有自己的消息循环和权限约束，完成后将结果返回给父 Agent。
