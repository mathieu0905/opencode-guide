Storage 存储层

源码位置：packages/opencode/src/storage/db.ts (数据库连接)
Schema：packages/opencode/src/storage/schema.ts (表定义)
迁移：packages/opencode/migration/

概述

OpenCode 使用 SQLite 作为嵌入式数据库，通过 Drizzle ORM 进行类型安全的查询。数据库文件存储在用户本地目录，实现了零配置的本地持久化。 最关键的设计是 Database.effect() 模式——确保事件在数据库事务提交后才发布。

数据库初始化 + 性能优化

// packages/opencode/src/storage/db.ts

import { Database as BunDatabase } from "bun:sqlite"
import { drizzle } from "drizzle-orm/bun-sqlite"

// 数据库路径
const dbPath = path.join(Global.Path.data, "opencode.db")
// → ~/.local/share/opencode/opencode.db

// 创建 SQLite 连接
const sqlite = new BunDatabase(dbPath)

// ═══════ 性能优化 PRAGMA ═══════

sqlite.run("PRAGMA journal_mode = WAL")
// ↑ Write-Ahead Logging: 读写不互相阻塞
//   默认的 DELETE 模式: 写入时锁定整个数据库, 读取被阻塞
//   WAL 模式: 写入到 WAL 文件, 读取看到 WAL 之前的快照
//   → Agent 写入 Part 数据时, 前端可以同时查询消息列表

sqlite.run("PRAGMA synchronous = NORMAL")
// ↑ 平衡性能与安全
//   FULL: 每次写入都 fsync → 最安全但最慢
//   NORMAL: 只在 checkpoint 时 fsync → 够安全且快
//   OFF: 不 fsync → 最快但断电可能丢数据

sqlite.run("PRAGMA busy_timeout = 5000")
// ↑ 并发写入时等待 5 秒 (不立即报错)

sqlite.run("PRAGMA cache_size = -64000")
// ↑ 64MB 页面缓存 (负数 = KB 单位)

sqlite.run("PRAGMA foreign_keys = ON")
// ↑ 启用外键约束 → 级联删除才会生效

const db = drizzle(sqlite)

各表详细 Schema

Session 表

SessionTable = sqliteTable("session", {
  id:                text("id").primaryKey(),
  //                 ↑ 降序 ULID (新的在前)
  project_id:        text("project_id").references(() => ProjectTable.id),
  parent_id:         text("parent_id"),       // Fork 父会话 (自引用)
  slug:              text("slug").notNull(),   // URL 友好 ID
  directory:         text("directory").notNull(),
  title:             text("title").notNull().default(""),
  version:           text("version").notNull().default(""),
  share_url:         text("share_url"),

  // ★ 反规范化字段 — 避免查询时遍历所有消息计算
  summary_additions: integer("summary_additions"),  // 新增行数
  summary_deletions: integer("summary_deletions"),  // 删除行数
  summary_files:     integer("summary_files"),      // 变更文件数
  summary_diffs:     text("summary_diffs", { mode: "json" }),

  revert:            text("revert", { mode: "json" }),
  permission:        text("permission", { mode: "json" }),

  time_created:      integer("time_created"),
  time_updated:      integer("time_updated"),
  time_compacting:   integer("time_compacting"),  // 非 null = 正在压缩
  time_archived:     integer("time_archived"),     // 非 null = 已归档
})

Message + Part 表

MessageTable = sqliteTable("message", {
  id:           text("id").primaryKey(),
  session_id:   text("session_id")
                  .references(() => SessionTable.id, { onDelete: "cascade" }),
  time_created: integer("time_created"),
  data:         text("data", { mode: "json" }),
  // ↑ 完整的 UserMessage | AssistantMessage JSON
  //   包含: role, agent, model, cost, tokens, error, finish...
})

PartTable = sqliteTable("part", {
  id:           text("id").primaryKey(),
  message_id:   text("message_id")
                  .references(() => MessageTable.id, { onDelete: "cascade" }),
  session_id:   text("session_id"),     // ★ 冗余字段
  //            ↑ 正规化设计不需要这个字段 (可以通过 message → session 查询)
  //            但 OpenCode 大量按 sessionID 查询 Part, 冗余避免了 JOIN
  time_created: integer("time_created"),
  data:         text("data", { mode: "json" }),
  // ↑ 12 种 Part 类型之一:
  //   TextPart | ToolPart | ReasoningPart | FilePart | SubtaskPart
  //   StepStartPart | StepFinishPart | SnapshotPart | PatchPart
  //   AgentPart | RetryPart | CompactionPart
})

WAL 模式工作原理


  ═══════ 默认模式 (DELETE/ROLLBACK) ═══════

  写入时:
    1. 复制原始页面到回滚日志
    2. 修改数据库文件中的页面
    3. 删除回滚日志 (完成提交)

  问题: 写入期间锁定整个数据库 → 读取被阻塞

  ─────────────────────────────────────────────────────

  ═══════ WAL 模式 (OpenCode 使用) ═══════

  ┌──────────────────────────────────────────────────┐
  │                                                  │
  │   数据库文件 (opencode.db)                        │
  │   ┌──────────────────────────────┐               │
  │   │ Page 1 │ Page 2 │ Page 3 │..│               │
  │   └──────────────────────────────┘               │
  │          ↑ 读取者看到这里的数据                    │
  │                                                  │
  │   WAL 文件 (opencode.db-wal)                      │
  │   ┌──────────────────────────────┐               │
  │   │ Frame 1: Page 2 (新版)       │               │
  │   │ Frame 2: Page 5 (新版)       │  ← 写入追加   │
  │   │ Frame 3: Page 2 (更新版)     │               │
  │   └──────────────────────────────┘               │
  │          ↑ 写入者追加到这里                        │
  │                                                  │
  │   结果:                                           │
  │     读取者: 看到 db 文件 → 不受写入影响           │
  │     写入者: 只追加到 WAL → 不修改 db 文件         │
  │     → 读写并行! ★                                │
  │                                                  │
  │   Checkpoint (自动):                              │
  │     WAL 文件的数据写回 db 文件 → WAL 清空         │
  │     通常在 WAL 超过 1000 页时自动触发             │
  └──────────────────────────────────────────────────┘

事务 + Database.effect() 模式


  ═══════ 为什么需要 effect()? ═══════

  场景: 创建 Session 后要广播事件, 前端收到后查询详情

  ❌ 错误做法:
  db.transaction(() => {
    db.insert(session)
    Bus.publish("session.created")  // ← 事件在事务内发布
    db.insert(message)              // ← 如果这里失败, 事务回滚
  })                                //    但事件已经发出! 前端看到了虚假事件

  ✅ 正确做法 (Database.effect):
  db.transaction(() => {
    db.insert(session)
    Database.effect(() => {
      Bus.publish("session.created")  // ← 推迟到事务提交后
    })
    db.insert(message)
    Database.effect(() => {
      Bus.publish("message.updated")  // ← 推迟到事务提交后
    })
  })
  // 事务提交成功 → 两个 effect 顺序执行
  // 事务回滚     → 两个 effect 丢弃, 不执行

  ═══════ 完整时序图 ═══════

  Database.transaction(fn)
       │
       ├── 1. BEGIN TRANSACTION
       │
       ├── 2. fn(tx) 执行:
       │       ├── tx.insert(session)        ← 数据写入
       │       ├── Database.effect(publish1)  ← 加入队列
       │       ├── tx.insert(message)        ← 数据写入
       │       └── Database.effect(publish2)  ← 加入队列
       │
       ├── 3. COMMIT  ← 所有数据原子写入
       │
       └── 4. flush effects:
               ├── publish1()  ← Bus.publish("session.created")
               └── publish2()  ← Bus.publish("message.updated")
                                  ↓
                             SSE → 前端
                             此时数据已确保存在 ✓

查询模式

// ═══════ 按 Session 查询所有消息 + Parts ═══════

const messages = db.select()
  .from(MessageTable)
  .where(eq(MessageTable.session_id, sessionID))
  .orderBy(asc(MessageTable.time_created))

const parts = db.select()
  .from(PartTable)
  .where(eq(PartTable.session_id, sessionID))
  // ★ 用冗余的 session_id 直接查, 不需要 JOIN message 表
  .orderBy(asc(PartTable.time_created))

// 组装成 WithParts 结构:
// [
//   { info: UserMessage, parts: [TextPart, FilePart] },
//   { info: AssistantMessage, parts: [TextPart, ToolPart, ToolPart, TextPart] },
// ]

// ═══════ 按时间查询最近 Session ═══════
// 降序 ID = 时间倒序 → 天然排序
const recent = db.select()
  .from(SessionTable)
  .where(and(
    eq(SessionTable.project_id, projectID),
    isNull(SessionTable.time_archived),  // 排除已归档
  ))
  .orderBy(asc(SessionTable.id))  // ID 升序 = 时间降序 (降序 ULID)
  .limit(20)

设计要点

  • 嵌入式 SQLite + WAL — 零配置部署，WAL 模式让读写互不阻塞。 Agent 高频写入 Part 数据时，前端查询不受影响。
  • Database.effect() 保证事件一致性 — 这是整个数据层最关键的设计。 事件一定在数据库事务提交后才发布，消除了前端"收到事件但查不到数据"的竞态条件。
  • JSON 列 + TypeScript 类型 — 12 种 Part 类型有不同的字段结构， 用 JSON 统一存储最灵活。新增 Part 类型无需 SQL 迁移。正确性由 TypeScript + Zod 在应用层保证。
  • 冗余字段换查询性能 — Part 表的 session_id 是冗余的（可以通过 message 表关联）， 但避免了高频查询中的 JOIN。Session 表的 summary_* 字段同理——反规范化换取 O(1) 查询。
  • 级联删除防孤儿数据 — SQLite 的 ON DELETE CASCADE 确保删除 Session 时 自动清理 Message → Part → Todo → Share，不会遗留任何垃圾数据。
