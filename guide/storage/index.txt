  ═══════ 为什么需要 effect()? ═══════

  场景: 创建 Session 后要广播事件, 前端收到后查询详情

  ❌ 错误做法:
  db.transaction(() => {
    db.insert(session)
    Bus.publish("session.created")  // ← 事件在事务内发布
    db.insert(message)              // ← 如果这里失败, 事务回滚
  })                                //    但事件已经发出! 前端看到了虚假事件

  ✅ 正确做法 (Database.effect):
  db.transaction(() => {
    db.insert(session)
    Database.effect(() => {
      Bus.publish("session.created")  // ← 推迟到事务提交后
    })
    db.insert(message)
    Database.effect(() => {
      Bus.publish("message.updated")  // ← 推迟到事务提交后
    })
  })
  // 事务提交成功 → 两个 effect 顺序执行
  // 事务回滚     → 两个 effect 丢弃, 不执行

  ═══════ 完整时序图 ═══════

  Database.transaction(fn)
       │
       ├── 1. BEGIN TRANSACTION
       │
       ├── 2. fn(tx) 执行:
       │       ├── tx.insert(session)        ← 数据写入
       │       ├── Database.effect(publish1)  ← 加入队列
       │       ├── tx.insert(message)        ← 数据写入
       │       └── Database.effect(publish2)  ← 加入队列
       │
       ├── 3. COMMIT  ← 所有数据原子写入
       │
       └── 4. flush effects:
               ├── publish1()  ← Bus.publish("session.created")
               └── publish2()  ← Bus.publish("message.updated")
                                  ↓
                             SSE → 前端
                             此时数据已确保存在 ✓
