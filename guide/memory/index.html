Memory & Context 管理

在长时间对话中，AI Agent 会面临 上下文窗口溢出 的问题。 OpenCode 通过 Compaction（压缩） 和 Pruning（修剪） 两种机制来管理上下文， 确保对话可以无限延续而不丢失关键信息。

为什么需要 Context 管理？

LLM 的上下文窗口是有限的：

  • Claude 3.5 Sonnet: 200K tokens
  • GPT-4: 128K tokens
  • Gemini 1.5 Pro: 2M tokens

当对话历史 + 工具输出超过这个限制时，模型会拒绝请求。 OpenCode 需要在不丢失关键信息的前提下，压缩或删除旧内容。

两种管理策略

1. Compaction（压缩）— 用摘要替换历史

当检测到上下文即将溢出时（达到 context_limit - reserved_buffer）， OpenCode 会触发 自动压缩：

  1. 创建一个特殊的 compaction 消息
  2. 让 AI 阅读整个对话历史，生成一个结构化摘要
  3. 摘要包含：目标、指令、发现、已完成工作、相关文件
  4. 后续对话中，用这个摘要替换被压缩的历史消息

压缩提示词模板

Provide a detailed prompt for continuing our conversation above.
Focus on information that would be helpful for continuing the conversation.

## Goal
[What goal(s) is the user trying to accomplish?]

## Instructions
- [What important instructions did the user give you]
- [If there is a plan or spec, include information about it]

## Discoveries
[What notable things were learned during this conversation]

## Accomplished
[What work has been completed, in progress, and left?]

## Relevant files / directories
[List of files that have been read, edited, or created]

触发条件

const COMPACTION_BUFFER = 20_000

// 当 token 使用量超过可用窗口时触发
const usable = model.limit.input - COMPACTION_BUFFER
if (tokens.total >= usable) {
  await SessionCompaction.process({ sessionID, auto: true })
}

2. Pruning（修剪）— 删除旧工具输出

工具调用的输出（如文件内容、命令结果）往往占用大量 tokens，但随着对话推进， 旧的工具输出可能不再相关。Pruning 会：

  • 从最新消息往回扫描，统计最近 40K tokens 的工具调用
  • 对于更早的工具调用，标记其输出为 compacted
  • 被标记的输出不会再发送给 LLM，节省上下文空间
  • 保护特殊工具（如 skill）不被修剪

修剪逻辑

const PRUNE_PROTECT = 40_000  // 保护最近 40K tokens
const PRUNE_MINIMUM = 20_000  // 至少修剪 20K 才执行

// 从后往前扫描，找到超过 40K 的旧工具输出
for (let i = messages.length - 1; i >= 0; i--) {
  for (const part of message.parts) {
    if (part.type === "tool" && part.state.status === "completed") {
      total += estimateTokens(part.state.output)
      if (total > PRUNE_PROTECT) {
        part.state.time.compacted = Date.now()  // 标记为已压缩
      }
    }
  }
}

配置选项

在 claude.json 中可以配置压缩行为：

{
  "compaction": {
    "auto": true,           // 是否自动触发压缩（默认 true）
    "prune": true,          // 是否启用修剪（默认 true）
    "reserved": 20000       // 预留 buffer（默认 20K tokens）
  },
  "agents": {
    "compaction": {
      "name": "compaction",
      "model": {
        "providerID": "anthropic",
        "modelID": "claude-3-5-sonnet-20241022"
      }
    }
  }
}

手动触发压缩

除了自动触发，用户也可以手动请求压缩：

POST /api/session/:sessionID/compact
{
  "auto": false  // 手动触发，不自动继续对话
}

Compaction vs Pruning 对比

特性Compaction（压缩）Pruning（修剪）
触发时机上下文接近满时每次对话前
处理对象整个对话历史旧的工具输出
处理方式生成 AI 摘要标记为已压缩，不发送
成本需要额外 LLM 调用无成本
信息保留保留关键信息完全删除

实现细节

压缩消息的存储

压缩生成的摘要会作为一条特殊的 assistant 消息存储：

{
  "role": "assistant",
  "mode": "compaction",
  "summary": true,
  "agent": "compaction",
  "parts": [
    {
      "type": "text",
      "text": "## Goal\n用户想要实现...\n\n## Accomplished\n已完成..."
    }
  ]
}

消息转换逻辑

在发送给 LLM 前，MessageV2.toModelMessages() 会：

  1. 检查是否有 summary: true 的消息
  2. 如果有，跳过该消息之前的所有历史
  3. 只发送摘要 + 摘要之后的消息
  4. 对于被 pruning 标记的工具输出，跳过其 output 字段

插件扩展点

插件可以通过 experimental.session.compacting hook 自定义压缩行为：

Plugin.on("experimental.session.compacting", async (event) => {
  return {
    context: [
      "Additional context to include in compaction prompt"
    ],
    prompt: "Custom compaction prompt template"
  }
})

最佳实践

  • 保持 reserved buffer 充足 — 默认 20K tokens 足够大多数场景
  • 定期手动压缩 — 在完成一个大任务后，手动触发压缩可以清理上下文
  • 使用大窗口模型 — Gemini 1.5 Pro 的 2M 窗口可以大幅减少压缩频率
  • 避免读取超大文件 — 单个文件超过 100K tokens 会快速耗尽上下文
