概述

Event Bus 是 OpenCode 的神经系统。所有内部状态变更——Session 创建、 消息更新、工具执行、权限请求——都通过 Bus 广播。前端通过 SSE 实时接收这些事件， 实现零轮询的实时 UI。理解事件系统，就理解了前后端是如何保持同步的。

Bus 实现 — 基于 EventEmitter

// bus/index.ts — 核心实现 (只有 106 行!)

// ★ 底层是 Node.js EventEmitter
const emitter = new EventEmitter()
emitter.setMaxListeners(0)  // 不限制监听器数量

// ═══════ 发布事件 ═══════
function publish<T>(definition: EventDefinition<T>, properties: T) {
  const event = {
    type: definition.type,    // e.g. "message.part.delta"
    properties,               // e.g. { sessionID, messageID, partID, field, delta }
  }

  // ★ 如果在事务中 → 延迟发布 (Database.effect)
  // ★ 如果不在事务中 → 立即发布
  emitter.emit(definition.type, event)
  emitter.emit("*", event)  // 通配符监听器
}

// ═══════ 订阅单个事件 ═══════
function subscribe<T>(
  definition: EventDefinition<T>,
  handler: (event: { type: string, properties: T }) => void
) {
  emitter.on(definition.type, handler)
  return () => emitter.off(definition.type, handler)  // 返回取消函数
}

// ═══════ 订阅所有事件 ═══════
function subscribeAll(handler: (event: AnyEvent) => void) {
  emitter.on("*", handler)
  return () => emitter.off("*", handler)
}

Database.effect() — 事件一致性的关键

这是整个事件系统最精妙的设计。问题场景：


  ❌ 没有 Database.effect() 时的竞态问题:

  时间线 ──────────────────────────────────────────────►

  后端:    [BEGIN TX]  [INSERT session]  [Bus.publish]  [COMMIT TX]
                                              │
                                              ▼
  前端:                                  收到事件! ──► GET /session/{id}
                                                              │
                                                              ▼
                                                        404! 数据还没提交!

  ─────────────────────────────────────────────────────────────

  ✅ 有 Database.effect() 时:

  时间线 ──────────────────────────────────────────────►

  后端:    [BEGIN TX]  [INSERT session]  [effect队列]  [COMMIT TX]  [flush效果]
                                                                        │
                                                                   Bus.publish
                                                                        │
                                                                        ▼
  前端:                                                            收到事件!
                                                                        │
                                                                        ▼
                                                                  GET /session/{id}
                                                                        │
                                                                        ▼
                                                                   200 OK! ✓

// 实现原理:

// Database.effect() 使用 AsyncLocalStorage 追踪当前事务上下文

const effectQueue = new AsyncLocalStorage<Array<() => void>>()

function effect(fn: () => void) {
  const queue = effectQueue.getStore()
  if (queue) {
    queue.push(fn)  // 在事务中 → 加入队列
  } else {
    fn()            // 不在事务中 → 立即执行
  }
}

function transaction(fn) {
  const queue: Array<() => void> = []

  effectQueue.run(queue, () => {
    db.transaction(() => {
      fn(db)  // 执行事务体 (内部的 effect 会加入 queue)
    })
    // ★ 事务已提交 → 安全地 flush 所有副作用
    for (const effect of queue) {
      effect()
    }
  })
}

// ═══════ 使用示例 ═══════
Database.transaction((tx) => {
  // 1. 写入数据
  tx.insert(SessionTable).values({ id: "ses_001", ... })

  // 2. 注册副作用 (此时不会执行)
  Database.effect(() => {
    Bus.publish(Session.Event.Created, { info: session })
  })

  // 3. 写入更多数据
  tx.insert(MessageTable).values({ ... })
})
// 4. 事务提交成功 → effect 中的 Bus.publish 才执行
// 5. 如果事务回滚 → effect 永远不会执行 → 不会发送虚假事件

applyDelta — 流式文本核心


  流式文本的前端累积过程:

  时间    Delta 事件              Store 中的 text        UI 显示
  ────────────────────────────────────────────────────────────
  T1     delta: "让"              "让"                   让▌
  T2     delta: "我看"            "让我看"               让我看▌
  T3     delta: "看代码"          "让我看看代码"          让我看看代码▌
  T4     delta: "..."             "让我看看代码..."       让我看看代码...▌
  T5     (part.updated)           "让我看看代码..."       让我看看代码...

  ─────────────────────────────────────────────────────────────

  // applyDelta 实现:
  function applyDelta(messageID, partID, field, delta) {
    const parts = this.parts[messageID]
    const idx = binarySearch(parts, partID)      // 二分查找
    const part = parts[idx]
    const current = part[field] ?? ""             // "让我看"
    const next = current + delta                  // "让我看" + "看代码" = "让我看看代码"
    parts[idx] = { ...part, [field]: next }       // 不可变更新
    // → Zustand 触发 React 重渲染
  }

  ═══════ 断线重连策略 (指数退避) ═══════

  第 1 次断线 → 1 秒后重连
  第 2 次断线 → 2 秒后重连
  第 3 次断线 → 4 秒后重连
  第 4+ 次断线 → 5 秒后重连 (上限)
  连接成功后 → 重置计数器

  ═══════ 心跳超时检测 ═══════
  如果 15 秒没收到任何数据 (包括 heartbeat):
    → 认为连接已断开
    → 主动关闭 EventSource
    → 触发重连流程

  │      Bus.publish("session.diff", { diffs })                │  ──► SSE ──► 前端更新 diff
  │  }                                                         │
  │                                                            │
  │  Bus.publish("message.updated", { info: { completed } })   │  ──► SSE ──► 前端停止 loading
  │  Bus.publish("session.idle", { sessionID })                │  ──► SSE ──► 前端状态恢复
  └────────────────────────────────────────────────────────────┘

设计要点

  • EventEmitter 足够了 — 不需要 Redis Pub/Sub 或 Kafka。 OpenCode 是单进程嵌入式应用，内存级 EventEmitter 延迟最低、最简单。
  • Database.effect() 是一致性保障 — 事件一定在数据写入成功后才发布。 事务回滚时副作用不会执行，前端永远不会收到"数据还不存在"的虚假事件。
  • 44 种事件覆盖所有状态变更 — 从 Session 创建到 Git 分支切换， 所有内部状态变化都有对应事件。前端可以构建完全响应式的 UI。
  • SSE 而非 WebSocket — SSE 是单向推送（服务端→客户端）， 恰好满足需求。比 WebSocket 更简单（浏览器原生支持、自动重连、无需握手协议）。 前端的操作（发消息、回复权限）走普通 HTTP POST。
  • GlobalEvent 的 directory 字段实现多项目隔离 — 一个 SSE 连接 接收所有项目的事件，前端按 directory 过滤。简单且无需为每个项目维护独立连接。
