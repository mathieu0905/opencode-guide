Message 消息模型

源码位置：packages/opencode/src/session/message-v2.ts (900 行)

概述

Message 是 Session 内的核心数据单元。但与 ChatGPT 等产品不同，OpenCode 的 Message 并不直接包含文本内容——它通过 Part 系统承载内容。

你可以这样理解：Message 是信封，Part 是信纸。 信封上写着「谁发的、什么时候、用什么模型、花了多少钱」， 信纸上才是实际的文字、工具调用、推理过程。 一个信封里可以装多张信纸——这就是为什么一条 AI 回复可以同时包含文本 + 多次工具调用 + 推理过程。

为什么要分成 Message 和 Part？

  • 流式更新 — AI 输出文字的同时可能触发工具调用。 分成 Part 后，可以独立更新每个 Part，甚至用 delta 只发送增量文本。
  • 工具状态追踪 — 工具从 pending → running → completed 的状态变化 独立于文本输出。Part 让每个工具调用有自己的状态机。
  • 类型安全 — 12 种 Part 有完全不同的字段。 联合类型比一个巨大的可选字段对象更安全也更清晰。

UserMessage — 用户消息

type UserMessage = {
  id: string
  sessionID: string
  role: "user"           // ★ 区分 User vs Assistant

  time: { created: number }

  agent: string          // 使用的 Agent 名称
  model: {
    providerID: string   // e.g. "anthropic"
    modelID: string      // e.g. "claude-opus-4"
  }

  // 可选覆盖
  format?: OutputFormat  // 让 AI 以 JSON 格式输出
  system?: string        // 临时覆盖 System Prompt
  tools?: Record<string, boolean>  // 工具开关
  variant?: string       // 模型变体

  summary?: {
    title?: string
    body?: string
    diffs: FileDiff[]    // 该消息导致的文件变更
  }
}

// UserMessage 的 Part 通常只有:
// - TextPart (用户输入的文字)
// - FilePart (用户上传的图片/文件)

AssistantMessage — AI 回复

type AssistantMessage = {
  id: string
  sessionID: string
  role: "assistant"
  parentID: string       // ★ 对应的 UserMessage ID

  time: {
    created: number      // 开始时间
    completed?: number   // 完成时间
    // completed 为 undefined = AI 仍在输出
    // 前端据此判断是否显示 loading
  }

  modelID: string        // 实际使用的模型
  providerID: string
  mode: string
  agent: string

  path: {
    cwd: string          // 当前工作目录
    root: string         // 项目根目录
  }

  // ===== 成本追踪 =====
  cost: number           // 总成本 (美元) e.g. 0.033
  tokens: {
    total?: number
    input: number        // 输入 token
    output: number       // 输出 token
    reasoning: number    // 推理 token (extended thinking)
    cache: {
      read: number       // 缓存命中 (省钱!)
      write: number      // 缓存写入
    }
  }

  error?: ProviderAuthError     // API Key 无效
       | UnknownError           // 未知错误
       | MessageAbortedError    // 用户中止
       | ContextOverflowError   // 上下文溢出
       | ...

  finish?: string        // "stop" | "tool_calls" | "length"
}

Part 系统 — 12 种内容类型

type Part =
  | TextPart          // 文本输出 — 最常见
  | ToolPart          // 工具调用 — 核心功能
  | ReasoningPart     // 推理过程 — Claude extended thinking
  | FilePart          // 文件附件 — 图片/PDF
  | SubtaskPart       // 子任务 — 启动子 Agent
  | StepStartPart     // 步骤开始标记
  | StepFinishPart    // 步骤结束 (含 token/成本统计)
  | SnapshotPart      // Git 快照
  | PatchPart         // 文件变更 Diff
  | AgentPart         // Agent 切换
  | RetryPart         // API 重试标记
  | CompactionPart    // 上下文压缩标记

TextPart — 文本

type TextPart = {
  id: string
  sessionID: string
  messageID: string
  type: "text"
  time: { created: number }

  text: string          // Markdown 内容
  synthetic?: boolean   // true = 系统生成 (非 LLM)
  ignored?: boolean     // true = 后续 LLM 调用不包含此 Part
}

// 流式更新:
// 创建时 text = "" → delta("Hello") → delta(" World") → delta("!")
// 最终 text = "Hello World!"

ToolPart + ToolState 状态机

type ToolPart = {
  id: string
  sessionID: string
  messageID: string
  type: "tool"
  time: { created: number }
  tool: string          // 工具名 e.g. "bash", "edit", "read"
  callID: string        // LLM 生成的调用 ID
  attachments: FilePart[]
  state: ToolState      // ★ 状态机
}

// 状态转换:
//   pending ──► running ──► completed
//                       └──► error

type ToolStatePending = {
  status: "pending"
  input: Record<string, unknown>
  // LLM 决定调用工具, 参数已解析, 尚未执行
  // 前端: 灰色卡片
}

type ToolStateRunning = {
  status: "running"
  input: Record<string, unknown>
  // 工具正在执行 (如 bash 命令运行中)
  // 前端: 黄色脉冲动画
}

type ToolStateCompleted = {
  status: "completed"
  input: Record<string, unknown>
  output: string        // 工具返回的文本结果
  title: string         // 显示标题 e.g. "Read src/app.ts (42 lines)"
  metadata: Record<string, unknown>
  // ★ output 发送回 LLM 作为 tool_result
  // 前端: 绿色卡片, 可展开查看 output
}

type ToolStateError = {
  status: "error"
  input: Record<string, unknown>
  error: string         // 错误信息
  // ★ 错误也发送回 LLM, 它可以据此调整策略
  // 前端: 红色卡片
}

StepStart / StepFinish — 步骤边界

Agent 循环的每一轮（一次 LLM 调用 + 工具执行）被 StepStart 和 StepFinish 包裹， 让前端可以知道「AI 的每一步做了什么、花了多少 token」：

type StepFinishPart = {
  type: "step-finish"
  tokens: {
    input: number
    output: number
    reasoning: number
    cache: { read: number, write: number }
  }
  cost: number     // 这一步的美元成本
  finish: string   // "stop" | "tool_calls" | "length"
}

// 示例: 一个包含 2 步的 AssistantMessage
//
// step-start ────── 第 1 步
//   text: "让我看看代码..."
//   tool: read(app.ts) → completed
// step-finish ───── (cost: $0.01)
//
// step-start ────── 第 2 步
//   text: "发现 bug, 修复中..."
//   tool: edit(app.ts) → completed
//   text: "已修复!"
// step-finish ───── (cost: $0.02)

其他 Part 类型

// ReasoningPart — Claude Extended Thinking
{ type: "reasoning", text: "..." }
// AI 的内部推理过程, 用户可展开查看

// FilePart — 文件附件
{ type: "file", mime: "image/png", filename: "screenshot.png", url: "data:..." }

// SubtaskPart — 子任务
{ type: "subtask", sessionID: "子Session的ID" }
// Task 工具创建子 Agent 时产生

// SnapshotPart — Git 快照
{ type: "snapshot", hash: "abc123..." }
// 记录文件在某时间点的状态, 用于 Revert

// PatchPart — 文件变更
{ type: "patch", hash: "...", files: [{ path: "app.ts", additions: 5, deletions: 2 }] }

流式更新机制 — Delta 详解

完整事件序列 — 一条文本回复

// 用户发送 "你好" 后, 前端收到的 SSE 事件:

// 1. AI 消息创建 (completed=undefined → 正在输出)
{ type: "message.updated", properties: { info: {
  id: "msg_002", role: "assistant", parentID: "msg_001",
  time: { created: 1706000000000 }
}}}

// 2. TextPart 创建 (空文本)
{ type: "message.part.updated", properties: { part: {
  id: "part_001", messageID: "msg_002", type: "text", text: ""
}}}

// 3-N. 逐 token 推送
{ type: "message.part.delta", properties: {
  sessionID: "ses_001", messageID: "msg_002", partID: "part_001",
  field: "text", delta: "你好"
}}
{ type: "message.part.delta", properties: { ..., delta: "！我是" }}
{ type: "message.part.delta", properties: { ..., delta: " AI 助手。" }}

// N+1. TextPart 最终状态
{ type: "message.part.updated", properties: { part: {
  id: "part_001", type: "text", text: "你好！我是 AI 助手。"
}}}

// N+2. 消息完成 (completed 有值了)
{ type: "message.updated", properties: { info: {
  id: "msg_002", time: { created: ..., completed: 1706000003000 },
  cost: 0.001, tokens: { input: 50, output: 20, ... }
}}}

前端如何处理 Delta

applyDelta(messageID, partID, field, delta) {
  const parts = this.parts[messageID]
  const idx = binaryFind(parts, partID)
  const part = parts[idx]
  const current = part[field] ?? ""    // "Hello"
  const next = current + delta          // "Hello World"
  parts[idx] = { ...part, [field]: next }
  // → Zustand 触发 React 重渲染
  // → 用户看到文字逐步出现
}

工具调用的事件序列

// AI 决定读取文件:

// 1. pending → 灰色卡片
{ type: "message.part.updated", properties: { part: {
  type: "tool", tool: "read",
  state: { status: "pending", input: { file_path: "/src/app.ts" } }
}}}

// 2. running → 黄色脉冲
{ type: "message.part.updated", properties: { part: {
  type: "tool", tool: "read",
  state: { status: "running", input: { file_path: "/src/app.ts" } }
}}}

// 3. completed → 绿色卡片, 可展开
{ type: "message.part.updated", properties: { part: {
  type: "tool", tool: "read",
  state: {
    status: "completed",
    input: { file_path: "/src/app.ts" },
    output: "import React from 'react'\n...",
    title: "Read src/app.ts (142 lines)",
    metadata: { lines: 142 }
  }
}}}

设计要点总结

  • Message + Part 二层结构 — 信封 + 信纸。 Message 记录元信息，Part 记录内容。 一条 AI 回复可以同时包含多种类型的 Part。
  • Delta 增量更新 — 流式输出的核心。 只发送新增的字符，前端本地追加，实现打字机效果。
  • ToolState 状态机 — pending → running → completed/error。 每次变化都通过 message.part.updated 推送，前端实时更新 UI。
  • StepStart/StepFinish — Agent 循环的步骤边界 + 成本统计。
  • JSON 存储 — 灵活（12 种 Part 不同字段）+ 无需 SQL 迁移 + TypeScript 类型保证。
