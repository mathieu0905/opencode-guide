Session 会话系统

源码位置：packages/opencode/src/session/index.ts (877 行)

概述

Session 是 OpenCode 的顶层组织单元。 你可以把它类比为 ChatGPT 左侧栏的一条对话记录，但 OpenCode 的 Session 功能更强大—— 它支持 Fork（从任意消息点分叉）、共享（生成公开链接）、归档和文件回滚。

每个 Session 关联到一个 Project（Git 仓库）和一个 Directory（工作目录）。 Session 内部包含多条 Message，每条 Message 又包含多个 Part。 这种三层结构（Session → Message → Part）是 OpenCode 数据模型的骨架。

Session 数据结构

下面是完整的 Session 类型定义，每个字段都有详细说明：

type Session = {
  // ===== 标识符 =====
  id: string
  // 降序 ULID — 越新创建的 Session，ID 越小 (字典序)
  // 这样 ORDER BY id ASC 就能得到时间倒序
  // 为什么用降序？大部分查询需要「最新的在前」

  slug: string
  // URL 友好的短标识，如 "fix-login-bug"

  projectID: string
  // 关联的 Project ID
  // 来自 Git 仓库的第一个 root commit hash
  // 同一个仓库的不同 clone 共享同一个 projectID

  directory: string
  // 工作目录的绝对路径

  parentID?: string
  // 父 Session 的 ID（仅 Fork 产生的子 Session 有值）
  // 形成树结构: 父 → 子 → 子子

  // ===== 展示信息 =====
  title: string        // AI 自动生成的标题
  version: string      // 版本标记

  // ===== 时间戳 =====
  time: {
    created: number     // 创建时间 (Unix ms)
    updated: number     // 最后更新时间
    compacting?: number // 非 null = 正在压缩上下文
    archived?: number   // 非 null = 已归档
  }

  // ===== 可选功能 =====
  summary?: {
    additions: number   // 新增行数
    deletions: number   // 删除行数
    files: number       // 变更文件数
    diffs?: FileDiff[]  // 详细 diff 列表
  }
  // 记录这个 Session 对项目做了哪些文件变更

  share?: { url: string }       // 共享链接
  permission?: PermissionRuleset // Session 级权限覆盖
  revert?: {                     // 文件回滚状态
    messageID: string
    partID?: string
    snapshot?: string
    diff?: string
  }
}

核心操作详解

1. 创建会话

// HTTP: POST /session
// SDK:  sdk.session.create({ title?, parentID?, permission? })
// 源码: session/index.ts → create()

async function create(input?) {
  // 第一步: 生成降序 ID
  // 降序 ULID: 时间越新, ID 字典序越小
  // 好处: ORDER BY id = 时间倒序, 查最新的不需要 DESC
  const id = descending()

  // 第二步: 生成 slug (URL 友好)
  const slug = generateSlug()

  // 第三步: 获取当前项目信息
  const project = Instance.project  // Git 仓库信息
  const directory = Instance.directory

  // 第四步: 写入数据库
  db.insert(SessionTable).values({
    id, slug,
    project_id: project.id,
    directory,
    title: input?.title ?? "",
    parent_id: input?.parentID,
    permission: input?.permission,
    time_created: Date.now(),
    time_updated: Date.now(),
  })

  // 第五步: 自动共享 (如果配置了 share: "auto")
  if (config.share === "auto") {
    await share(id)
  }

  // 第六步: 广播事件
  // ★ 在 Database.effect() 中执行 → 数据库事务提交后才广播
  // 防止: 前端收到事件后查询数据库, 但数据还没写入
  Bus.publish(Event.Created, { info: session })

  return session
}

2. Fork 会话 — 从历史分叉

想象你和 AI 对话了 10 轮，第 5 轮的回答不理想，你想从第 4 轮重新开始—— 这就是 Fork。它创建一个新 Session，复制前 4 轮的消息，你在新 Session 中继续。

// HTTP: POST /session/{id}/fork
async function fork(sessionID, messageID, options?) {
  // 1. 创建子 Session (parentID 指向原 Session)
  const child = await create({ parentID: sessionID })

  // 2. 复制消息 (直到 messageID)
  const allMessages = await messages(sessionID)
  for (const msg of allMessages) {
    if (msg.info.id > messageID) break  // ID 降序, 大 = 更早
    // 复制 Message + Parts 到子 Session
    insert(child.id, msg)
  }

  // 3. 可选: 恢复文件状态到 Fork 点
  if (options?.revert) {
    await Snapshot.restore(snapshotHash)
  }

  return child
}

// Fork 后的数据关系:
// 原 Session:  Msg1 → Msg2 → Msg3 → Msg4(Fork点) → Msg5 → Msg6
// 子 Session:  Msg1'→ Msg2'→ Msg3'→ Msg4'→ (用户继续新对话)

子 Session 与状态持久化

当 LLM 调用 Task 工具时，会创建一个子 Session—— 它是数据库里一条独立的 Session 记录，parentID 指向父 Session。 子 Session 有自己的消息流，独立于父 Session。

这个设计在进程崩溃恢复时至关重要。 Session、Message、Part 都持久化在 SQLite 中，进程重启后完好无损。 但 SessionStatus（busy/idle）是纯内存状态：

// status.ts — 纯内存，不持久化
const state = Instance.state(() => {
  const data: Record<string, Info> = {}
  return data
})

export function get(sessionID: string) {
  return state()[sessionID] ?? { type: "idle" }
  //                            ^^^^^^^^^^^^^^^^
  //                            不在 Map 里 = idle
}

// 进程崩溃前: "session_123" → { type: "busy" }
// 进程重启后: Map 是空的 → 所有 session 默认 "idle"
// → 不会出现"永远 busy"的 session

为什么不持久化？因为 SessionStatus 反映的是当前进程的执行状态。 进程死了，就不存在"正在执行"的概念。让它自然回到 idle 是正确的行为。 更多恢复细节见 Agent 执行循环 的"崩溃恢复"部分。

设计要点

为什么用降序 ID？

大部分 UI 需要「最新的会话在最前面」。降序 ULID 让默认排序就是时间倒序。 分页也更简单：WHERE id > cursor LIMIT 20 就能取下一页。

为什么 summary 存在 Session 上？

这是 反规范化 (denormalization)——用冗余数据换取查询性能。 前端直接显示「+42 -17 3 files」而不需要遍历所有消息重新计算。 每次 Agent 产生文件变更时，summary 通过 session.diff 事件实时更新。

级联删除的重要性

SQLite 的 ON DELETE CASCADE 确保删除 Session 时自动清理所有关联数据 （Message → Part → Todo → SessionShare → 子 Session）。 没有这个机制，遗漏任何一步都会产生孤儿数据。
